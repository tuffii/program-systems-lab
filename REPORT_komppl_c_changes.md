# Отчёт по изменениям в `src/komppl.c`

Документ составлен по **полному `git diff`** между закоммиченной версией файла и текущим рабочим деревом (без коммита). Базовый коммит: `fd4630e` → текущее состояние (`443 insertions / 340 deletions` по статистике diff).

---

## Как устроен пайплайн (куда попадают правки)

Кратко, в терминах этого компилятора:

1. **Ввод** — исходный `.pli` читается в `ISXTXT[]` (80-байтные карточки).
2. **Лексика** — `compress_ISXTXT()` → сжатая строка `STROKA`.
3. **Синтаксис** — `sint_ANAL()` по таблицам `SINT[]`, `VXOD[]`, `TPR[][]` → стек достижений `DST[]`.
4. **Семантика + генерация ASM** — `gen_COD()` вызывает функции из таблицы `FUN[][]` (1-й и 2-й проходы); при генерации заполняется `ASS_CARD`, карточки сбрасываются в `ASSTXT[]` через `ZKARD()`.
5. **Вывод** — `PRO2()` пишет `ASSTXT` в файл `.ass` (`fwrite` по 80 байт).

Правки в `komppl.c` затрагивают в основном **п. 3–4** (грамматика и семантика) и **формат выходной карточки** (поле комментария `COMM`).

## 2. Грамматика: таблица `SINT[]` и связи `ALT`

### 2.1. Ветка `IPE` → альтернатива с `RZR`

- **Правило `SINT[168]`**: поле `ALT` изменено с `0` на `229`.
- **Смысл**: после разбора `IPE` (идентификатор в выражении) допускается альтернативный путь, где вторым операндом выступает **числовой литерал `RZR`** (например `5` в `B * 5`).
- **Пайплайн**: синтаксис.

### 2.2. Ветка объявления `DCL … C` → `DEC FIXED`

- **Правило `SINT[187]`**: `ALT` изменён с `0` на `201`.
- **Добавлены правила `SINT[201]…[225]`**: последовательность лексем для `DEC FIXED (RZR)` с ветками с/без `INIT`, вызов `ODC` и т.д. (зеркально к уже существующей ветке `BIN FIXED`).
- **Смысл**: парсинг объявлений **упакованного десятичного** типа `DEC FIXED`.
- **Пайплайн**: синтаксис → затем `ODC1()` на 1-м проходе.

### 2.3. Символ `=` как знак операции `ZNK` (сравнение)

- **Добавлены `SINT[226]…[228]`**: вводится цепочка `=  →  ZNK`.
- **Смысл**: в правой части присваивания допускается конструкция вида `C = A = C` (сравнение даёт булево значение).
- **Пайплайн**: синтаксис → `AVI2()` на 2-м проходе.

### 2.4. Литерал `RZR` как часть `AVI`

- **Добавлены `SINT[229]…[231]`**: `RZR` в контексте `AVI`.
- **Смысл**: завершение грамматики для использования константы в арифметическом/смешанном выражении.
- **Пайплайн**: синтаксис.

## 5. Комментарии в ASM-карточке: `COMM_LIM`, `SETCOMM()`

| Изменение | Суть | Зачем | Пайплайн |
|-----------|------|-------|----------|
| `#define COMM_LIM 25` | Ограничение длины текста комментария в поле `COMM`. | Короткие, читаемые пояснения без разрастания карточки. | Генерация (`ASS_CARD` → `ZKARD`) |
| `SETCOMM(const char *s)` | Запись комментария в `ASS_CARD._BUFCARD.COMM` с учётом **переполнения операнда**: если длинная строка операнда уже «залезла» в область `COMM`, сохраняется её **префикс (хвост операнда)**, затем (при необходимости) **разделитель-пробел**, и только потом текст комментария. | Иначе при `memcpy(ASS_CARD.BUFCARD+15, …)` длинные операнды визуально **слипаются** с комментарием (`…@FIVE(1)Mul…`). | Генерация |

Вызовы `SETCOMM(...)` расставлены перед соответствующими `ZKARD()` в `OPR2`, `AVI2`, `OPA2`, `OEN2` (см. разделы ниже).

## 10. `AVI2()` — генерация кода для выражений (ядро EX11)

Заменена прежняя логика с операндами `RRAB` и комментариями «Load variable…» / «Form intermediate value…» на целевую схему под задание.

### 10.1. Один терм (`IFORMT == 1`)

| Тип в `SYM` | Генерация | Комментарий (`SETCOMM`) | Результат |
|-------------|-----------|-------------------------|-----------|
| `'B'` (BIN FIXED) | `LH @R2,<имя>` | краткое пояснение загрузки | `RESULT_REG = "@R2"` |
| `'D'` (DEC FIXED) | `ZAP @BUF(8),<имя>(<RAZR>)` через `memcpy(BUFCARD+15,…)` | пояснение копирования в буфер | (результат дальше в цепочке `*`) |

### 10.2. Операция `*` (после ZAP из левой части)

| Действие | Инструкции | Дополнительно |
|----------|------------|----------------|
| Умножение на литерал | `MP` с операндом `@BUF(8),@…(1)` | `const_label()` + запись в `EXTRA[]` при первом появлении значения |
| Двоичный результат | `CVB @R1,@BUF` | `RESULT_REG = "@R1"` |

### 10.3. Операция `=` (сравнение, булево)

| Последовательность | Назначение |
|--------------------|------------|
| `CH @R2,<правый идентификатор>` | сравнение с тем, что уже в `@R2` (после загрузки `A`) |
| `BC 8,@TRUE` | переход при равенстве |
| `LA @R3,0` / `B @SAVE` / `@TRUE: LA @R3,1` | ветвление false/true |
| `PENDING_LABEL = "@SAVE"` | метка для следующей `STH` |
| `RESULT_REG = "@R3"` | булево в `@R3` |

### 10.4. `+` / `-` (для BIN FIXED, два терма)

Сохранена идея `AH`/`SH` с операндом `@R2,<второй идентификатор>`, без старых `RRAB` и длинных комментариев из оригинала.

**Пайплайн:** семантика 2-й проход, непосредственно **генерация ASM** (`ASS_CARD` + `ZKARD`).

## 12. `OEN2()` — эпилог программы (данные, EQU, END)

Крупная переработка относительно исходника.

| Блок | Было (идея) | Стало | Пайплайн |
|------|-------------|-------|----------|
| Возврат | `BCR` с операндом `15,14`, метки регистров без `@` | `BCR 15,@RVIX` + `SETCOMM` | Генерация |
| Переменные `SYM` | Только `TYPE=='B'`, всегда `DC`, старые комментарии | Пропуск не-буквенных имён и типов не `B`/`D`; для `B`: `DS H` если нет `INIT`, иначе `DC H'…'`; для `D`: `DC PL…`; `memset` метки + корректная длина имени | Генерация секции данных |
| Константы из `EXTRA` | не было | Цикл `DC PL…` для каждой записи | Генерация |
| Буфер и выравнивание | не было | При `NEXTRA>0`: `DS 0F`, затем `@BUF DC PL8'0'` | Генерация |
| EQU | `RBASE`, `RRAB` без `@` | `@RBASE`, `@R1`, `@R2`, `@R3`, `@RVIX` с нужными номерами | Генерация |
| `END` | копирование имени программы в операнд (и риск UB с `i++`) | Только `END` без операнда + `SETCOMM("Program ends")` | Генерация |

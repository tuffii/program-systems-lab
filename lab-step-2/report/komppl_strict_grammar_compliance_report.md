# Отчет о строгом приведении `komppl.c` к документации

## Цель правки

Исправлялись три несоответствия, найденные в предыдущей версии `lab-step-2/src/komppl.c`:

- оператор `C=A=C;` не должен разбираться как арифметическое присваивание `OPA -> IPE = AVI ;`;
- знак `=` не должен быть вариантом `<ZNK>`, так как по документации `<ZNK> ::= + | - | *`, а `<ZNL> ::= =`;
- генерация сравнения должна выполняться из семантики `<AVL>`, а не из `AVI2()` по специальной проверке `op_sign == '='`.

Итог: текущая версия транслирует `examppl.pli` и `ex11.pli`, при этом логическое присваивание представлено через цепочку `<OPL> -> <IPE> = <AVL> ;`, а логическое сравнение через `<AVL> -> <AVI> <ZNL> <IPE>`.

## Сравнение с `lab-step-1`

Статистика отличий:

```text
{lab-step-1 => lab-step-2}/src/komppl.c | 1223 +++++++++++++++++++------------
1 file changed, 748 insertions(+), 475 deletions(-)
```

Основные причины отличий:

- расширены размеры таблиц `NSINT`, `NVXOD`, `NNETRM`, `NSTROKA`, `MAXLTXT`;
- добавлены нетерминалы `LID`, `AVL`, `OPL`, `ZNL`;
- добавлены ветви `DCL ... INIT(<LID>)` для `BIN FIXED` и `DEC FIXED`;
- добавлены пары семантических методов `LID1/LID2`, `AVL1/AVL2`, `OPL1/OPL2`, `ZNL1/ZNL2`;
- доработан синтаксический анализатор для корректного выбора ветви `OPA`/`OPL` при общем префиксе `IPE = ...`.

## Исправленные несоответствия

### 1. Реальный маршрут `C=A=C;`

Теперь логическое присваивание не сворачивается через `OPA` с арифметическим `AVI`.

В таблице `SINT` присутствуют документированные ветви:

- `TEL OPL -> TEL`: строки `270-272`;
- `AVI ZNL LIT -> AVL`: строки `283-286`;
- `AVI ZNL LID -> AVL`: строки `283`, `288-289`;
- `AVI ZNL IPE -> AVL`: строки `283`, `291-292`;
- `IPE = AVL ; -> OPL`: строки `294-298`.

Также сохранена техническая цепочка `302-305` под уже разобранным общим префиксом `IPE = ...`. Она не вводит новую грамматику, а представляет ту же продукцию `IPE = AVL ; -> OPL` в форме, удобной для существующего табличного анализатора. Проверка показала, что без этой цепочки `ex11.pli` снова не разбирается, потому что исходный алгоритм отката альтернатив не возвращает указатель входа на начало оператора после частично выбранной ветви `OPA`.

### 2. Знак `=` исключен из `<ZNK>`

В текущей версии:

- терминал `"=  "` в `VXOD` указывает на куст `ZNL`;
- старые строки `226-228`, где раньше было `"=  " -> "ZNK"`, очищены;
- в строке терминала `=` таблицы `TPR` больше нет связи с `ZNK`, оставлена только связь с `ZNL`;
- в исходном коде больше нет ветки `op_sign == '='`.

Это соответствует документации:

- продукция 54: `+ -> <ZNK>`;
- продукция 55: `- -> <ZNK>`;
- продукция 56: `* -> <ZNK>`;
- продукция 57: `= -> <ZNL>`.

### 3. Генерация сравнения перенесена в `AVL2()`

В `AVI2()` удалена специальная обработка `op_sign == '='`.

Теперь генерация кода сравнения выполняется из:

```c
int AVL2 ()
{
  FORM ();
  if ( IFORMT < 2 )
    return 2;

  avl2_emit_compare_rhs ( FORMT [IFORMT - 1] );
  return 0;
}
```

Метод `OPL2()` отвечает только за сохранение результата логического выражения в левую переменную. Это разделяет роли согласно документации:

- `<AVL>` формирует результат логического выражения;
- `<OPL>` выполняет логическое присваивание.

## Изменения в синтаксическом анализаторе

Добавлены две небольшие функции просмотра вперед:

- `is_logical_assignment(pos)` считает количество знаков `=` до ближайшей `;`;
- `has_equal_before_semicolon(pos)` проверяет, есть ли логический знак `=` внутри правой части до `;`.

Они используются только для разрешения конфликта общего префикса:

- `OPA` и `OPL` начинаются с `IPE =`;
- внутри `AVL` левая часть начинается как обычный `AVI`.

Это не меняет грамматику, а помогает существующему табличному анализатору выбрать документированную ветвь до того, как он уйдет в неверную альтернативу и потеряет позицию отката.

## Соответствие документации

| Документация | Реализация в `lab-step-2/src/komppl.c` | Статус |
|---|---|---|
| `<LID> ::= <CIF> | <LID><CIF>` | `SINT` строки `300-301`, `273-276`; `FUN` содержит `LID1/LID2` | соответствует |
| `<AVI> ::= <LIT> | <LID> | <IPE> | <AVI><ZNK>...` | сохранены арифметические ветви, `ZNK` содержит только `+`, `-`, `*` | соответствует |
| `<AVL> ::= <AVI><ZNL><LIT/LID/IPE>` | `SINT` строки `283-292`, генерация в `AVL2()` | соответствует |
| `<OPL> ::= <IPE>=<AVL>;` | `SINT` строки `294-298`, сохранение в `OPL2()` | соответствует |
| `<ZNL> ::= =` | `VXOD "=" -> 280`, `SINT` строки `280-282` | соответствует |
| `<TEL> ::= ... | <TEL><OPL>` | `SINT` строки `270-272`, альтернатива от `TEL OPA` к `TEL OPL` | соответствует |

## Проверка по итерациям

Тестирование выполнялось в Docker-окружении `lab32` с 32-битной сборкой:

```bash
gcc -m32 -o build/komppl.exe komppl.c
./build/komppl.exe input/ex11.pli
./build/komppl.exe input/examppl.pli
```

Итерации:

- после удаления `=` из `ZNK` и первичного подключения `ZNL`: `examppl.pli` проходил, `ex11.pli` падал на откате альтернатив;
- после выбора `OPL` на уровне `TEL`: `ex11.pli` все еще падал, потому что конфликт оставался внутри правой части после `IPE =`;
- после подключения выбора `AVL` внутри общего префикса `IPE = ...`: оба файла прошли;
- после очистки старой ветви `= -> ZNK` и удаления отладочного вывода: оба файла прошли;
- попытка убрать техническую цепочку `302-305` показала повторное падение `ex11.pli`, поэтому цепочка возвращена как представление той же продукции `IPE = AVL ; -> OPL` для текущего алгоритма разбора.

Финальный результат:

```text
ex11_status:0
examppl_status:0
ex11_vs_lab_step_1_1_output:0
ex11_vs_etalon:0
examppl_vs_etalon:0
```

Код собирается с прежними предупреждениями исходной программы о `return` без значения в `main` и форматах `%s`; эти предупреждения не относятся к внесенной правке.

## Вывод

Текущая версия `lab-step-2/src/komppl.c` устраняет ранее зафиксированные структурные несоответствия:

- `C=A=C;` обрабатывается как логическое присваивание через `OPL/AVL/ZNL`;
- `=` больше не является вариантом `ZNK`;
- `avl2_emit_compare_rhs()` вызывается из `AVL2()`.

Сгенерированные `ex11.ass` и `examppl.ass` совпадают с эталонными файлами.

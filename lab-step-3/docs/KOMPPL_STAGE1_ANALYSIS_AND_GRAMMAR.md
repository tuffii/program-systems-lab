# Первый этап компилятора (`komppl.c`): анализ и модификация БД синтаксического анализатора

Документ описывает, как устроен первый этап в `lab-step-3/src/komppl.c` (в задании фигурирует имя `comppl.c` — имеется в виду этот модуль), разбивает его на подэтапы, фиксирует план адаптации под программу **EX11** с сохранением работоспособности **EXAMP**, и даёт согласованную с требованиями **модифицированную грамматику** и перечень изменений в «базе данных» распознавателя (таблицы `SINT`, `VXOD`, матрица `TPR`, семантические действия). Реализация в коде **не включена**.

---

## 1. Назначение файла `komppl.c`

Файл объединяет в одном проходе по конвейеру:

1. **Лексический уровень** — сжатие исходного текста.
2. **Синтаксический уровень** — разбор по табличной грамматике с возвратами (backtracking).
3. **Семантический уровень** — двухпроходная интерпретация стека достижений (`DST`) с генерацией текста ассемблера.

То есть это не отдельный «только парсер», а **фронтенд + первая фаза генерации кода** для ограниченного подмножества PL/I.

---

## 2. Подэтапы первого этапа (по порядку выполнения в `main`)

### 2.1. Ввод исходного текста

- Строки читаются в массив `ISXTXT` (до `MAXNISXTXT` записей по 80 символов).
- Имя файла передаётся аргументом командной строки; расширение должно быть `.pli`.
- Базовое имя без расширения сохраняется в `NFIL` для формирования выходного `.ass`.

### 2.2. Лексический анализ — `compress_ISXTXT()`

**Роль:** превратить исходник в **одну сплошную строку** `STROKA` без «лишних» пробелов, с фиксированными правилами вокруг разделителей.

**Идея:**

- Пробелы после `;`, `:`, `(`, `)` и между повторяющимися пробелами удаляются или нормализуются.
- Перед символами `+ - = * ( )` лишний пробел перед оператором убирается (сдвиг `I3--`), чтобы цепочка шла компактно: например `D=A+B-C;`.
- Символ `PREDSYM` запоминает предыдущий значимый символ для решения, нужен ли пробел.

**Важно для расширения языка:** новые односимвольные лексемы (если появятся) нужно вписать в те же правила «склейки» с пробелами, иначе сжатие разорвёт лексему.

### 2.3. Матрица допустимых сочетаний — `TPR` и `build_TPR()`

**Роль:** задать **граф достижимости** между «входными символами» (терминалы + нетерминалы из таблицы `VXOD`) в смысле: «после разбора фрагмента с символом A может ли начинаться разбор B».

- Исходная матрица смежности `TPR` заполняется вручную (см. комментарии в коде: строки нетерминалов `AVI`…`ZNK` и терминалов `A`…`*`).
- `build_TPR()` применяет алгоритм Уоршелла и **транзитивно замыкает** достижимость: если из `X` можно попасть в `Y`, а из `Y` в `Z`, то из `X` в `Z`.

**Зачем:** ускорить/направить разбор в `sint_ANAL`: проверки `TPR[...][...]` отсекают заведомо неверные ветви.

**При расширении грамматики:** для **каждого нового нетерминала** (`LVI`, `LID`, `OPL`, `ZNL` и т.д.) нужно:

- добавить столец/строку в матрицу (сейчас `NNETRM == 16` — потребуется увеличить размер и пересобрать все строки `TPR`);
- явно проставить связи: кто может следовать за кем в сжатой строке.

### 2.4. Таблица правил в форме распознавания — `SINT[]`

**Роль:** задать **детерминированное дерево альтернатив** для нисходящего разбора.

Каждая запись:

- `POSL` — ссылка на следующий элемент правила;
- `PRED` — предыдущий;
- `DER` — терминал (один значащий символ в первой позиции, остальное пробелы) **или** имя нетерминала (3 буквы + пробел);
- `ALT` — альтернативная ветка при неуспехе.

Особый маркер: правило с `DER == "*  "` означает **успешное завершение** разбора нетерминала (лист с `*` в следующей позиции в коде).

Корень программы: нетерминал `PRO`, стартовая запись в `SINT` с индексом `1` (после фиктивной строки `0`).

**При расширении:** каждую новую продукцию нужно разложить на цепочку записей `SINT` с корректными `ALT` для разветвлений.

### 2.5. Таблица входов в «кусты» правил — `VXOD[]`

**Роль:** для символа (терминала или нетерминала) указать **индекс первой записи** в `SINT`, с которой начинать разбор (`VX`), и тип: `'T'` — терминал, `'N'` — нетерминал.

**При расширении:** для каждого нового нетерминала — новая строка в `VXOD` и начальная позиция в `SINT`.

### 2.6. Синтаксический анализ — `sint_ANAL()`

**Роль:** построить **правый вывод** в виде последовательности записей в стеке достижений `DST[]`, используя стек целей `CEL[]`.

**Алгоритм (упрощённо):**

1. Кладётся цель `PRO` с позиции `I=0` в `STROKA`.
2. Для текущего символа входа берётся стартовый индекс из `VXOD`.
3. Идёт обход списка `SINT`:  
   - если встретился терминал — сравнивается с `STROKA[I]`;  
   - если нетерминал — кладётся новая цель в `CEL`;
   - при успешном завершении нетерминала (`*`) в `DST` записывается фрагмент: имя нетерминала, **левая и правая границы** подстроки в `STROKA` для последующей семантики.
4. При неуспехе — переход по `ALT` или откат по `PRED`.

**Коды ошибок:** `0` — успех; `1` — с первого символа нельзя начать `PRO` (нет пути в `TPR`); `2` — полный отказ (вернулись к начальной цели с `CEL3 == 999`).

### 2.7. Семантика — `gen_COD()` и пары `*1()` / `*2()`

**Роль:** обойти `DST` **дважды** и для каждого нетерминала вызвать функцию из таблицы `FUN[NNETRM][2]`:

- **1-й проход (`*1`)** — в основном наполнение таблицы имён `SYM[]` (объявления, проверки), подготовка данных.
- **2-й проход (`*2`)** — генерация псевдо-ассемблера в `ASSTXT` через `ZKARD()`.

Связка с фрагментом текста: `FORM()` выделяет из `STROKA` подстроку по `DST2`…`DST4` и режет её на «лексемы» фиксированной ширины в `FORMT[][]` (массив строк по 9 символов).

**Текущие типы в `SYM[].TYPE` (по `ODC1`, `AVI2`, …):**

- `'P'` — имя программы (процедура);
- `'B'` — `BIN FIXED`;
- `'U'` — неизвестный/ошибочный тип (для не-`BIN` сейчас ошибка).

Для **EX11** понадобятся новые коды (например, `'D'` для `DEC FIXED`) и ветки в семантике упакованных операций — это уже за пределами «чистого» синтаксиса, но таблица `SYM` должна их предусмотреть.

## 3. Что уже покрывает текущая грамматика в коде (связь с EXAMP)

Пример **EXAMP** в терминах реализованных конструкций:

- Пролог: `IPR : PROC OPTIONS ( MAIN );` → нетерминалы `OPR`, `OPR1`/`OPR2`.
- Объявления: `DCL … BIN FIXED ( RZR ) [ INIT ( LIT ) ];` где `LIT` — **двоичный** литерал `MAN` + `B` (`VALUE()` в коде переводит двоичную строку в число).
- Тело: чередование `ODC` и `OPA`; `TEL` рекурсивен.
- Оператор присваивания: `IPE = AVI ;` с `AVI`, использующим `ZNK` (`+`, `-`, `*`, `=`).
- Эпилог: `END IPR`.

**Ограничение:** в таблице `SINT` ветка `ODC` жёстко завязана на `BIN` и фиксированную структуру лексем в `FORMT` (номера полей в `ODC1`).

## 4. Целевая программа EX11 и требование совместимости

Исходный текст (смысл задания):

```text
EX11: PROC OPTIONS ( MAIN );
DCL A BIN FIXED INIT ( 3 );
DCL B DEC FIXED INIT ( 3 );
DCL C BIN FIXED;
C = B * 5;
C = A = C;
END EX11;
```

Ожидаемое поведение компилятора:

- Первая строка присваивания — **арифметика** с десятичной константой `5`, умножением на `B` (packed decimal), результат в `C`.
- Вторая — **логическое сравнение** `A` и `C`, результат (0/1) в `C` (цепочка присваивания с внутренним `=` — именно **логический** `=`, не «оператор сложения»).

**Совместимость с EXAMP:** все старые конструкции (обязательная разрядность `( RZR )`, двоичные `INIT ( …B )`, арифметика с `ZNK` без логического `=`) должны оставаться в грамматике как **отдельные альтернативы** в `ODC`, `AVI`, `TEL`, `OPA`/`OPL`.

## 5. План адаптации `komppl.c` (без кода, по слоям)

### 5.1. Лексика

- Убедиться, что ключевое слово `DEC` и константы без суффикса `B` корректно попадают в `STROKA` (после сжатия — как непрерывные цепочки букв/цифр; пробелы вокруг `(` `)` уже обрабатываются).
- При появлении новых ключевых буквенных последовательностей при необходимости расширить нормализацию (если понадобится отличать токены — в текущей модели многие ключи разбираются посимвольно в `SINT`).

### 5.2. Таблицы грамматики

1. Увеличить `NNETRM`, `NSINT`, `NVXOD` с запасом под новые нетерминалы: минимум **`LID`, `LVI`, `OPL`, `ZNL`** (и при необходимости вспомогательные).
2. Добавить строки в `VXOD` и корни правил в `SINT` для:
   - новых вариантов `ODC` (без `RZR`, `DEC`, `INIT` с `LID`);
   - `OPL` и `LVI`;
   - разделения `ZNK` (без `=`) и `ZNL` (только `=`).
3. Перестроить матрицу `TPR`: новые столбцы/строки, затем снова `build_TPR()`.

### 5.3. Семантика

- **`ODC1`:** разбор полей через `FORM()` потребует **согласованной нумерации** слотов в `FORMT` для всех вариантов `DCL` (варианты с/без `( RZR )`, `BIN`/`DEC`, `INIT ( LIT )` / `INIT ( LID )` / без `INIT`). Для каждого варианта — отдельная ветка или общий разбор с флагами.
- **`SYM`:** хранить тип `'B'` / `'D'`, разрядность (в т.ч. «по умолчанию»), инициализацию в строковом виде (`3B` vs `3` для packed/decimal).
- **`AVI2` / новый `LVI2`:**  
  - `AVI` — только `ZNK` ∈ `{+, -, *}` и при необходимости вложенный `=` **исключить** (перенос на `LVI`).  
  - `LVI` — цепочки с `ZNL` (`=`) для сравнения; результат — булево для генерации ветвления (как в эталонном `.ass`).
- **`OPA2` vs `OPL2`:** разные шаблоны кода (арифметика vs сравнение и присвоение 0/1).

### 5.4. Порядок внедрения (рекомендуемый)

1. Расширить грамматику и `SINT` так, чтобы **парсер принимал** и EXAMP, и EX11 (даже с заглушками в семантике).
2. Включить объявления `DEC` и литералы `LID` в `SYM`.
3. Разделить семантику присваиваний `OPA` / `OPL`.
4. Реализовать генерацию под packed decimal и сравнение (согласно вашему эталону ASS).

## 6. Модифицированная грамматика (требования учтены)

Ниже — целевая **БНФ** с явным разделением арифметического и логического `=`, отдельным десятичным литералом, оператором `OPL`, нетерминалом `LVI` и знаком `ZNL`. Символ `_` в терминалах — **пробел** (как в исходном задании).

### 6.1. Программа и оболочка

| Правило | Форма |
|--------|--------|
| ⟨PRO⟩ | ::= ⟨OPR⟩⟨TEL⟩⟨OEN⟩ |
| ⟨OPR⟩ | ::= ⟨IPR⟩:PROC_OPTIONS(MAIN); |
| ⟨IPR⟩ | ::= ⟨IDE⟩ |
| ⟨IDE⟩ | ::= ⟨BUK⟩ \| ⟨IDE⟩⟨BUK⟩ \| ⟨IDE⟩⟨CIF⟩ |
| ⟨BUK⟩ | ::= A \| B \| C \| D \| E \| M \| P \| X |
| ⟨CIF⟩ | ::= 0 \| 1 \| … \| 9 |
| ⟨OEN⟩ | ::= END_⟨IPR⟩ |

*Замечание:* для имён вроде `EX11` уже используется ⟨IDE⟩ с ⟨CIF⟩; набор ⟨BUK⟩ можно при необходимости расширить под лабораторию (например, буква `I` для `EX11` — если в вашем алфавите она должна быть разрешена, добавьте её в ⟨BUK⟩).

### 6.2. Тело программы

**Требование:** в ⟨TEL⟩ поддержать логический оператор присваивания и цепочки с `ZNL`.

| Правило | Форма |
|--------|--------|
| ⟨TEL⟩ | ::= ⟨ODC⟩ \| ⟨TEL⟩⟨ODC⟩ \| ⟨TEL⟩⟨OPA⟩ \| ⟨TEL⟩⟨OPL⟩ |

### 6.3. Объявления ⟨ODC⟩ (п. 8): разрядность опциональна, BIN и DEC

Ввести вспомогательное понятие **описатель типа** с необязательной разрядностью.

Обозначим абстрактно (для компактности):

- `FIXED_HEAD_BIN` → последовательность токенов `BIN_FIXED` с опциональной `( RZR )`.
- `FIXED_HEAD_DEC` → `DEC_FIXED` с опциональной `( RZR )`.

**Литералы инициализации:**

- ⟨LIT⟩ — двоичный, как раньше: ⟨MAN⟩B.
- ⟨LID⟩ — **десятичный** целочисленный литерал (без суффикса `B`): удобно определить как ⟨RZR⟩ (одна или более цифр), что совпадает с примерами `3`, `5`, `10`.

| Правило | Форма |
|--------|--------|
| ⟨ODC⟩ | ::= DCL_⟨IPE⟩_BIN_FIXED \[_(⟨RZR⟩)_\]_; |
| | \| DCL_⟨IPE⟩_BIN_FIXED \[_(⟨RZR⟩)_\]_INIT_(⟨LIT⟩)_; |
| | \| DCL_⟨IPE⟩_BIN_FIXED \[_(⟨RZR⟩)_\]_INIT_(⟨LID⟩)_; |
| | \| DCL_⟨IPE⟩_DEC_FIXED \[_(⟨RZR⟩)_\]_; |
| | \| DCL_⟨IPE⟩_DEC_FIXED \[_(⟨RZR⟩)_\]_INIT_(⟨LID⟩)_; |

Квадратные скобки здесь — метаязык «ноль или один раз». В таблице `SINT` это оформляется **двумя или более альтернативными ветками** `ALT` для каждой комбинации наличия `( RZR )` и `INIT`.

**Сохранение EXAMP:** остаются ветки с обязательным `( RZR )` и `INIT ( LIT )` с двоичным литералом, как в текущей реализации.

| ⟨RZR⟩ | ::= ⟨CIF⟩ \| ⟨RZR⟩⟨CIF⟩ |
| ⟨LIT⟩ | ::= ⟨MAN⟩B |
| ⟨LID⟩ | ::= ⟨RZR⟩ |
| ⟨MAN⟩ | ::= 1 \| ⟨MAN⟩0 \| ⟨MAN⟩1 |

*Примечание:* если потребуются ведущие нули в ⟨LID⟩ (например `03`), они покрываются ⟨RZR⟩ как цепочкой цифр.

### 6.4. Присваивание: арифметическое ⟨OPA⟩ и логическое ⟨OPL⟩ (п. 13)

| Правило | Форма |
|--------|--------|
| ⟨OPA⟩ | ::= ⟨IPE⟩=⟨AVI⟩; |
| ⟨OPL⟩ | ::= ⟨IPE⟩=⟨LVI⟩; |

### 6.5. Арифметическое выражение ⟨AVI⟩

**Требование (п. 15):** из ⟨ZNK⟩ убрать `=`; равенство остаётся только в ⟨ZNL⟩.

| ⟨AVI⟩ | ::= ⟨LIT⟩ \| ⟨LID⟩ \| ⟨IPE⟩ \| ⟨RZR⟩ |
| | \| ⟨AVI⟩⟨ZNK⟩⟨LIT⟩ \| ⟨AVI⟩⟨ZNK⟩⟨LID⟩ \| ⟨AVI⟩⟨ZNK⟩⟨IPE⟩ \| ⟨AVI⟩⟨ZNK⟩⟨RZR⟩ |
| ⟨ZNK⟩ | ::= + \| - \| * |

*Пояснение для EX11:* в выражении `B * 5` символ `5` может трактоваться как ⟨RZR⟩ (как в вашей исходной схеме) или как ⟨LID⟩; оба согласованы с семантикой «положительное целое». В коде генерации важно согласовать с эталоном ASS (константа `@FIVE` и т.д.).

### 6.6. Логическое выражение ⟨LVI⟩ и знак ⟨ZNL⟩ (п. 15, п. 7)

| ⟨LVI⟩ | ::= ⟨LIT⟩ \| ⟨LID⟩ \| ⟨IPE⟩ \| ⟨RZR⟩ |
| | \| ⟨LVI⟩⟨ZNL⟩⟨LIT⟩ \| ⟨LVI⟩⟨ZNL⟩⟨LID⟩ \| ⟨LVI⟩⟨ZNL⟩⟨IPE⟩ \| ⟨LVI⟩⟨ZNL⟩⟨RZR⟩ |
| ⟨ZNL⟩ | ::= = |

**Смысл:** единственная «логическая» операция на этом этапе — **сравнение на равенство**; цепочка `A = C` задаёт вычисление булева значения; внешнее `C = …` оформляется как ⟨OPL⟩.

**Неразличимость символа `=` на лексическом уровне:** один и тот же символ `=` в тексте различается **контекстом грамматики** (после ⟨OPL⟩/⟨LVI⟩ идёт ⟨ZNL⟩, внутри ⟨AVI⟩ — недопустим).

## 7. Соответствие новых нетерминалов таблицам в `komppl.c`

| Нетерминал | Назначение | Где отражается |
|------------|------------|----------------|
| `LID` | Десятичный литерал (INIT и операнды, где нужно) | `VXOD`, `SINT`, столбец/строка `TPR`, `FUN`: `LID1`, `LID2` |
| `LVI` | Логическое (сравнительное) выражение | то же: `LVI1`, `LVI2` |
| `OPL` | Оператор логического присваивания | `OPL1`, `OPL2`; в `TEL` альтернатива рядом с `OPA` |
| `ZNL` | Логическое равенство | `ZNL1`, `ZNL2`; в правилах для `LVI` вместо `ZNK` |

Константа `NNETRM` и массив `FUN` должны быть расширены; индексы в `numb()` для всех нетерминалов должны остаться согласованными с `VXOD`.

## 8. Итоговый чек-лист требований

| Требование | Решение в грамматике |
|------------|----------------------|
| Отдельно от ⟨LIT⟩ для двоичных — десятичный литерал | Введён ⟨LID⟩ ::= ⟨RZR⟩ |
| Отдельно от арифметического `=` — логический | ⟨ZNK⟩ без `=`; ⟨ZNL⟩ ::= `=` |
| DCL без разрядности, BIN/DEC, INIT (3) | Расширен ⟨ODC⟩ с опциональным `( RZR )`, ветками BIN/DEC, INIT с ⟨LIT⟩ или ⟨LID⟩ |
| Оператор ⟨OPL⟩ в п. 13 | ⟨OPL⟩ ::= ⟨IPE⟩=⟨LVI⟩; |
| ⟨AVI⟩ заменяется на ⟨LVI⟩ в логическом присваивании | ⟨OPL⟩ использует ⟨LVI⟩; ⟨OPA⟩ оставляет ⟨AVI⟩ |
| В ⟨TEL⟩ учтён логический след | ⟨TEL⟩ включает ⟨TEL⟩⟨OPL⟩ |

## 9. Связь с эталонным ASS (кратко)

Эталон задаёт:

- Работу с **packed decimal** для `B`, умножение на константу 5, конвертацию в двоичное полу-слово для `C`.
- Ветвление по сравнению `A` и `C` с записью булева результата в `C`.

Полная генерация — во **2-м проходе** семантики (`*2` функции) и выходит за рамки «только БД синтаксиса», но **грамматика выше** задаёт ровно те деревья разбора, под которые эта генерация пишется.

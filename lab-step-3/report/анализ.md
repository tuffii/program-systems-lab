## 1. Разбор каждой операции в `ex11.ass`

Ниже приведено подробное описание каждой строки ассемблерного кода из файла `ex11.ass` (эквивалент личного задания).

*   `START 0` — Директива компилятора. Означает начало программы и инициализирует счетчик относительного адреса (`CHADR`) нулем.
*   `BALR @RBASE,0` — (Branch and Link Register). Загружает адрес следующей инструкции в базовый регистр `@RBASE` (регистр 5). Поскольку второй операнд равен 0, переход не выполняется, происходит только сохранение адреса.
*   `USING *,@RBASE` — Директива ассемблера. Указывает компилятору, что начиная с текущего адреса (обозначается `*`), для вычисления смещений следует использовать регистр `@RBASE` (5) в качестве базового.
*   `ZAP @BUF(8),B(3)` — (Zero and Add Packed). Команда формата SS (Storage-to-Storage). Очищает 8-байтовый буфер `@BUF` и копирует в него 3-байтовое упакованное десятичное значение переменной `B`.
*   `MP @BUF(8),@FIVE(1)` — (Multiply Packed). Команда формата SS. Умножает упакованное десятичное значение в буфере `@BUF` (длина 8 байт) на константу `@FIVE` (длина 1 байт). Результат сохраняется в `@BUF`.
*   `CVB @R1,@BUF` — (Convert to Binary). Команда формата RX. Конвертирует упакованное десятичное число из памяти (`@BUF`) в двоичное целое число и сохраняет его в регистр `@R1` (регистр 6).
*   `STH @R1,C` — (Store Halfword). Команда формата RX. Сохраняет младшие 2 байта (полуслово) из регистра `@R1` в память по адресу переменной `C`.
*   `LH @R2,A` — (Load Halfword). Команда формата RX. Загружает 2-байтовое значение (полуслово) из памяти (переменная `A`) в регистр `@R2` (регистр 2), расширяя знак до 4 байт.
*   `CH @R2,C` — (Compare Halfword). Команда формата RX. Сравнивает значение в регистре `@R2` с 2-байтовым значением в памяти по адресу `C`.
*   `BC 8,@TRUE` — (Branch on Condition). Команда формата RX. Условный переход. Маска `8` означает переход, если результат предыдущего сравнения "равно" (Equal). Если равно, прыгаем на метку `@TRUE`.
*   `LA @R3,0` — (Load Address). Команда формата RX. Загружает значение `0` в регистр `@R3`. В данном контексте это означает присваивание логического значения `False`.
*   `B @SAVE` — Безусловный переход на метку `@SAVE`. (Подробнее про команду `B` см. ниже).
*   `@TRUE LA @R3,1` — Метка `@TRUE`. Загружает значение `1` в регистр `@R3` (логическое `True`).
*   `@SAVE STH @R3,C` — Метка `@SAVE`. Сохраняет результат логического выражения (0 или 1) из регистра `@R3` в переменную `C`.
*   `BCR 15,@RVIX` — (Branch on Condition Register). Безусловный возврат из подпрограммы. Маска `15` означает "переход при любом условии". Адрес возврата находится в регистре `@RVIX` (14).
*   `A DC H'3'` — (Define Constant). Выделяет 2 байта памяти (Halfword) и инициализирует их значением 3.
*   `B DC PL3'3'` — (Define Constant). Выделяет 3 байта памяти в упакованном десятичном формате (Packed Decimal) и инициализирует их значением 3.
*   `C DS H` — (Define Storage). Резервирует 2 байта памяти (Halfword) без инициализации.
*   `@FIVE DC PL1'5'` — Выделяет 1 байт в упакованном десятичном формате со значением 5.
*   `DS 0F` — Выравнивание счетчика адреса на границу полного слова (Fullword, кратно 4). `0` означает, что память не выделяется, просто сдвигается счетчик `CHADR`.
*   `@BUF DC PL8'0'` — Резервирует 8 байт под рабочий буфер в упакованном формате со значением 0.
*   `@RBASE EQU 5` — (Equate). Директива. Связывает символическое имя `@RBASE` с числовым значением 5 (номер регистра).

## 2. Объяснение команды `B` и маски 15

В отчете есть следующая строка:
> `{"B", 0x47, 4, FRX, SRX}` (обрабатывается как RX, но с неявным первым операндом-маской 15)

**Что это значит?**
1. В архитектуре IBM 370 **нет отдельной машинной команды безусловного перехода** (команды `B`).
2. Вместо нее используется команда условного перехода `BC` (Branch on Condition), которая имеет код операции `0x47`.
3. Формат команды `BC` выглядит так: `BC M1, D2(X2,B2)`, где `M1` — это 4-битная маска условия.
4. Маска условия проверяет регистр состояния (Condition Code). Если маска равна `15` (в двоичном виде `1111`), это означает "переход при любом условии" (т.е. безусловный переход).
5. Ассемблерная мнемоника `B метка` — это просто "синтаксический сахар" (удобное сокращение) для программиста. Компилятор транслирует `B метка` точно так же, как если бы было написано `BC 15,метка`.
6. Поэтому в таблице `TMOP` мы указываем для команды `B` тот же код операции `0x47` и те же обработчики `FRX` и `SRX`, что и для команды `BC`. Но внутри функции `SRX` (на втором проходе) компилятор должен понять, что если встретилась мнемоника `B`, то первый операнд (маску) не нужно искать в тексте, а нужно жестко подставить число `15`.

## 3. Подробное объяснение таблицы ручного ассемблирования

Таблица ручного ассемблирования демонстрирует работу компилятора, который функционирует в **два прохода (просмотра)**.

### 3.1. Как рассчитываются относительные адреса (Первый проход)
**Главная цель 1-го прохода:** Вычислить адреса всех меток и сформировать Таблицу символов (`TSYM`).

Компилятор читает код сверху вниз. У него есть счетчик адреса (`CHADR`), который изначально равен 0. Встречая команду, компилятор прибавляет к `CHADR` длину этой команды:
*   Команды формата **RR** (например, `BALR`, `BCR`) занимают **2 байта**.
*   Команды формата **RX** (например, `CVB`, `STH`, `LH`, `LA`, `B`) занимают **4 байта**.
*   Команды формата **SS** (например, `ZAP`, `MP`) занимают **6 байт**.
*   Директивы выделения памяти (`DC`, `DS`) занимают столько байт, сколько указано в типе (например, `H` = 2 байта, `PL3` = 3 байта).
*   Директивы компилятора (`START`, `USING`, `EQU`) занимают **0 байт** (они не генерируют машинный код).

**Пример шагов 1-го прохода:**
1. `START 0` $\rightarrow$ `CHADR` = 0.
2. `BALR @RBASE,0` $\rightarrow$ Текущий адрес **0**. Длина команды 2 байта. Новый `CHADR` = 0 + 2 = **2**.
3. `USING *,@RBASE` $\rightarrow$ Текущий адрес **2**. Длина 0 байт. Новый `CHADR` остается **2**.
4. `ZAP @BUF(8),B(3)` $\rightarrow$ Текущий адрес **2**. Длина команды 6 байт. Новый `CHADR` = 2 + 6 = **8**.
5. `MP @BUF(8),@FIVE(1)` $\rightarrow$ Текущий адрес **8**. Длина команды 6 байт. Новый `CHADR` = 8 + 6 = **14**.

Встречая метку (например, `A`, `B`, `@TRUE`), компилятор просто запоминает текущее значение `CHADR` как адрес этой метки.
*   *Почему в операндах стоят знаки вопроса `?`* На первом проходе компилятор еще не знает адреса меток, которые объявлены ниже по коду. Например, обрабатывая `LH @R2,A`, компилятор еще не дошел до строки `A DC H'3'` и не знает адрес метки `A`. Поэтому вместо смещений ставятся `?`.

### 3.2. Откуда берется базовый адрес "2" и почему мы его вычитаем?
В архитектуре IBM 370 используется **базирование**. Процессор вычисляет физический адрес в памяти во время выполнения программы по формуле:
**`Физический Адрес = Значение_Базового_Регистра + Смещение`**

Компилятор должен сформировать машинный код, в котором записано именно **Смещение** (Displacement), а не сам физический адрес. Чтобы найти Смещение, компилятор решает простое уравнение:
**`Смещение = Физический Адрес - Значение_Базового_Регистра`**

Мы **не "двигаемся назад"** с помощью минуса. Мы просто находим разницу (расстояние) между нужной нам переменной и началом нашей базы.

**Откуда взялась цифра 2?**
1. В строке 2 написана команда: `BALR @RBASE,0`. Она находится по относительному адресу `0` и занимает `2` байта.
2. Команда `BALR` (Branch and Link Register) при выполнении процессором делает следующее: она берет адрес *следующей* за ней инструкции и кладет его в указанный регистр (`@RBASE`).
3. Следующая инструкция находится по адресу `0 + 2 = 2`. Значит, во время выполнения программы в регистре `@RBASE` будет лежать число **2**.
4. В строке 3 написана директива: `USING *,@RBASE`. Звездочка `*` означает "текущий адрес компиляции" (который сейчас равен 2). Эта директива говорит компилятору: *"Запомни, что во время выполнения в регистре `@RBASE` (регистр 5) будет лежать базовый адрес 2. Используй это для расчета смещений"*.

### 3.3. Подробный разбор вычисления операндов (Второй проход)
**Главная цель 2-го прохода:** Подставить реальные адреса вместо знаков вопроса и сформировать финальный операнд в формате `Смещение(Индекс, База)`.

Индексный регистр у нас не используется, поэтому он всегда равен `0`. Базовый регистр `@RBASE` равен `5`. Базовый адрес равен `2`.

**Пример 1: Команда `LH @R2,A` (строка 8)**
*   Компилятор смотрит в Таблицу символов: адрес метки `A` равен **52**.
*   Вычисляем смещение: `Смещение = Адрес_A - Базовый_адрес = 52 - 2 = 50`.
*   Формируем операнд: `Смещение(Индекс, База)` $\rightarrow$ **`50(0,5)`**.
*   *Чтение результата:* "Возьми значение из регистра 5, прибавь к нему 50, и по полученному адресу (52) прочитай данные".

**Пример 2: Команда `ZAP @BUF(8),B(3)` (строка 4)**
*   Это команда формата SS, у нее два операнда, указывающих на память.
*   Смотрим в Таблицу символов: адрес `@BUF` равен **64**, адрес `B` равен **54**.
*   Вычисляем смещение для первого операнда: `Смещение_BUF = 64 - 2 = 62`.
*   Вычисляем смещение для второго операнда: `Смещение_B = 54 - 2 = 52`.
*   Формируем операнды (в формате SS также указываются длины): `Смещение1(Длина1, База1), Смещение2(Длина2, База2)`.
*   Результат: **`62(8,5), 52(3,5)`**.

**Пример 3: Команда `B @SAVE` (строка 12)**
*   Смотрим в Таблицу символов: адрес метки `@SAVE` равен **46**.
*   Вычисляем смещение: `Смещение = 46 - 2 = 44`.
*   Формируем операнд: **`44(0,5)`**. (Напоминаем, что маска 15 здесь подразумевается неявно).
